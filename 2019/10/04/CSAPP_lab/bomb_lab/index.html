实验结果展示！

实验介绍

Dr. Evil是个bad guy，他为了毁灭世界而做了一个炸弹程序，善良且优秀的我们需要把炸弹拆除。

  • 程序分为6个部分（除了表面的6部分，其实还有一个小彩蛋。）。
  • 每个部分读入一个字符串。
  • 字符串不符合程序的要求，程序就会“爆炸”！

代码结构分析

因为D.Evil的疏忽大意，他泄漏了描述bomb程序main函数的bomb.c文件。

光靠这个.c文件显然是不行的，所以我们需要将bomb反汇编，LINUX $: objdump -d bomb > bomb.d，汇编语言摆在面前时，所有的真相都在里面等着我们探索。

剩下的工作就只剩下

  1. 阅读汇编代码
  2. 研究程序功能（可以整理成高级语言或者绘制流程图，当然也可以硬着头皮想（我就硬着头皮想出了所有答案，还发现了彩蛋，最后在写这些字的时候才把这个思考过程规范化。不建议硬想，伤身体，要善用工具。））
  3. 利用gdb调试

gdb调试技巧

本部分的介绍可以先跳过，需要的时候再看。

  1. 功能指令：
  • break 函数名 在函数处添加断点：例如 break phase_1。
  • break *0x0x80483c3 在固定地址添加断点。
  • delete 可以删除所有断点，加上参数n可以删除第n个断点。
  • run < filename 开始执行，把filename文件当作程序输入。
  • Continue 继续执行（会把之后的指令都执行完，除非遇到断点）。
  • stepi 按照指令流顺序执行一条指令。
  • stepi n n是一个数字，执行n条。
  • nexti 和stepi类似，但是！nexti会跳过函数调用。也可加参数n使用。
  • disass 将部分程序反汇编并打印，不加参数打印当前函数，可以加参数指定地址、指定函数等等。
  • print /x $rsp 以十六进制(/x)的格式打印寄存器rsp的内容。
  • pirnt /x *(long*) 0x402010 打印存在0x402010地址处的long数值。
  • x/s 0x402010 打印存在0x402010的字符串(/s)。*print的地址要加*但是x不需要加*。
  • 以上的指令都可以用 help 指令名字 来查看帮助。例: help print。
  • run可以简写为r，continue～c，stepi～si，nexti~ni等等。
  1. 我的调试经验总结：
  • 如果在挑战phase_1，先break phase_1。
  • 然后输入r < myanwser.txt。
  • 程序这时会停在phase_1。
  • 利用print和x搜集寄存器以及内存的信息。
  • 利用si、ni来验证程序功能是否符合自己的思路就可以了。
  • 忘记指令的位置，可以用disass查看当前位置。

思路

这个函数看起来是一个很复杂的递归调用，但是其实是，因为有一个调用返回值必会大于1，而其他的调用返回无法把这个1带来的影响消除，所以，函数永远不会走那条路。所以函数就变成了“一条路走到黑”再返回，我们输入的值决定它走到哪里返回，就变得很简单。

综上，答案不唯一，可以是“{7, 3, 1, 0}(任选一个) 0”，比如 7 0。

secret_phase_solution

.c文件中，Dr.Evil就提示我们有彩蛋，在我们看反汇编代码的时候，可能会发现，phase_6后面有一个secret_phase和fun7。

如何执行secret_phase?

在bomb反汇编文件内搜索secret_phase出现的位置，发现其在explode_defused函数中出现，而且如果满足一定的条件就会执行。

条件：

  1. 输入的字符行数num_of_string应该等于6。（在read_line读取一行输入时，全局变量num_of_string会+1）
  2. 两个字符串相等。

用gdb查看这两个字符串可以得到:”Dr.Evil”和一个看不懂的字符串。这两个字符串一定和我们的输入有关，如果无关，这个条件判断的结果就永远不会变，而这个看不懂的字符串恰好就是我在phase_4的输入，破案了！
